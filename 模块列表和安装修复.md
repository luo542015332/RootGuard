## 问题 1：模块列表显示非模块文件

### 问题描述
模块列表显示了非模块文件（如 `README.md`、`README.md.sha256` 等），而不是真正的模块目录。

### 根本原因
`MagiskProvider.kt` 中的 `getModules()` 方法使用 `ls -1` 命令列出 `/data/adb/modules` 目录的内容，这会返回所有文件和目录，包括文件。

### 修复方案
将 `ls -1` 改为 `find -type d`，只列出目录：

```kotlin
// 修复前
val process = Runtime.getRuntime().exec("su -c ls -1 $MODULES_PATH")
val moduleDirs = process.inputStream.bufferedReader().readLines()

// 修复后
val process = Runtime.getRuntime().exec("su -c find $MODULES_PATH -mindepth 1 -maxdepth 1 -type d")
val moduleDirs = process.inputStream.bufferedReader().readLines()
moduleDirs.forEach { dirPath ->
    val moduleId = dirPath.substringAfterLast("/")
    // ...
}
```

## 问题 2：模块安装失败无详细日志

### 问题描述
模块安装失败时没有详细的错误信息，难以定位问题。

### 根本原因
`installKernelSUModule()` 方法中：
1. 使用 `Runtime.getRuntime().exec(String)` 执行命令，没有正确处理带空格的参数
2. 错误信息没有被记录到日志
3. `cp -r $tempDir/*` 在某些 shell 下可能不工作（如果文件名包含通配符字符）

### 修复方案

1. **改用数组形式执行命令**，避免 shell 解析问题：
```kotlin
// 修复前
Runtime.getRuntime().exec("su -c unzip -o \"$zipPath\" -d $tempDir")

// 修复后
Runtime.getRuntime().exec(arrayOf("su", "-c", "unzip", "-o", zipPath, "-d", tempDir))
```

2. **添加详细的调试日志**，记录每个步骤的状态：
```kotlin
Logger.d("Starting KernelSU module installation from: $zipPath")
Logger.d("Creating temp directory: $tempDir")
Logger.d("Extracting zip file to temp directory")
Logger.d("Module ID: $moduleId")
// ...
```

3. **记录错误输出**：
```kotlin
if (extractProcess.exitValue() != 0) {
    val error = extractProcess.errorStream.bufferedReader().readText()
    Logger.e("Failed to extract module zip, exit code: ${extractProcess.exitValue()}, error: $error")
}
```

4. **在 module.prop 检查失败时列出目录内容**，帮助调试：
```kotlin
if (moduleProp.exitValue() != 0) {
    Logger.e("No module.prop found in $tempDir, not a valid module")
    val lsProcess = Runtime.getRuntime().exec(arrayOf("su", "-c", "ls", "-la", tempDir))
    val lsOutput = lsProcess.inputStream.bufferedReader().readText()
    Logger.d("Temp directory contents: $lsOutput")
}
```

5. **修复 cp 命令**，使用 `.` 代替 `*`：
```kotlin
// 修复前
Runtime.getRuntime().exec("su -c cp -r $tempDir/* $modulePath/")

// 修复后
Runtime.getRuntime().exec(arrayOf("su", "-c", "cp", "-r", "$tempDir/.", modulePath))
```

## 测试建议

### 测试模块列表
1. 安装更新后的 APK
2. 进入模块页面
3. 检查是否只显示真正的模块目录，不再有 README.md 等文件

### 测试模块安装
1. 准备一个有效的 Magisk/KernelSU 模块 zip 文件
2. 在 PandaSU 中安装该模块
3. 检查日志页面，应该能看到详细的安装过程日志：
   - "Starting KernelSU module installation from: ..."
   - "Creating temp directory: ..."
   - "Extracting zip file to temp directory"
   - "Module ID: ..."
   - "KernelSU module installed successfully at: ..." 或具体的错误信息

### 测试场景

#### 成功场景
- 使用标准的 Magisk 模块 zip 文件
- 模块包含有效的 module.prop 文件
- 应该能看到完整的安装日志并成功安装

#### 失败场景及日志
1. **无效的 zip 文件**
   - 日志：`Failed to extract module zip, exit code: X, error: ...`

2. **缺少 module.prop**
   - 日志：`No module.prop found in ..., not a valid module`
   - 日志：`Temp directory contents: ...`（列出实际内容）

3. **权限问题**
   - 日志：`Failed to create temp directory` 或 `Failed to copy files`

## 已知限制

1. **KernelSU 模块兼容性**：某些模块（如 Shamiko）会检查 KernelSU 的集成方式（子模块 vs 复制源代码）。如果用户的 KernelSU 不是通过子模块方式集成，这些模块会拒绝安装。这是模块本身的限制，不是 PandaSU 的 bug。

2. **建议**：对于有版本检查的复杂模块，建议使用 KernelSU Manager 安装。
